Persistent Queue Project

Overview

A producer-consumer system with a persistent queue: producers submit jobs (file paths), consumers process them (add timestamps), managed by Supervisor, with Streamlit UI for monitoring and control.

Approach

  • SQLite Queue: Persistent storage, atomic operations for job exclusivity.
  • Supervisor: Manages multiple producers/consumers, auto-restarts on crashes.
  • Streamlit: Simple UI for manager (resubmit/cancel) and ops (status).
  • Python: Core logic in `producer.py`, `consumer.py`, `sqlite_queue.py`.

Components & Choices

PersistentQSQLite

What: SQLite-based queue implementation.
Why: Lightweight, file-based persistence, atomic transactions.
Instead of: Redis (complex setup), in-memory (no persistence).

Producer

What: Generates random files every 5s, submits to queue.
Why: Simulates job creation.
Instead of: Manual input (less dynamic).

Consumer

What: Processes jobs (adds timestamps), 7-15s delay.
Why: Mimics real-world processing lag.
Instead of: Instant (unrealistic).

Supervisor

What: Runs 1 producer, 2 consumers.
Why: Robust process management, crash recovery.
Instead of: Manual scripts (no auto-restart), Honcho (less control).

Streamlit

What: Manager UI (resubmit/cancel), Ops UI (status table).
Why: Quick, interactive web interface.
Instead of: FastAPI (overkill for local UI), Flask (more setup).

Requirements.txt

What: Dependency list.
Why: Simple, standard.
Instead of: Poetry (extra complexity).
